# spec/03_constraints.md

## 1. System Constraints

### 1.1 Deterministic Core (Non-Negotiable)

The system MUST operate as a fully deterministic engine.

#### Requirements

* Same inputs MUST always produce identical outputs
* No randomness, heuristics, or probabilistic behavior allowed

#### Enforcement

* All state MUST be derived from events
* No hidden state or side effects

---

### 1.2 Event Model Constraints

#### Must

* Events MUST be append-only
* Events MUST be immutable after insertion
* Each event MUST be uniquely identified by `event_id`

#### Must Not

* Update existing events
* Delete events
* Modify timestamps post-write

#### Trade-off

* Increased storage usage vs guaranteed auditability and determinism

---

### 1.3 Idempotency Constraints

#### Must

* Duplicate requests MUST NOT create duplicate records
* System MUST tolerate retries safely

#### Enforcement

* `event_id = lead_id + ":" + section`
* Enforced via PRIMARY KEY at DB level

#### Escalation Trigger

* Duplicate events detected > 0 → immediate investigation

---

### 1.4 Section Ordering Constraints

#### Must

* Section order MUST be defined explicitly in `course_registry.py`
* Progression MUST use index-based lookup

#### Must Not

* Sort sections
* Infer order via string comparison
* Dynamically compute ordering

#### Risk

* Incorrect ordering breaks progression logic

---

### 1.5 TOTAL_SECTIONS Constraint

#### Must

* Defined ONLY in `execution/course/course_registry.py`

#### Must Not

* Be duplicated
* Be hardcoded elsewhere
* Be passed as parameter

#### Enforcement

* Imported wherever needed

#### Escalation Trigger

* Multiple definitions detected

---

### 1.6 Time Constraints

#### Must

* All timestamps MUST be UTC
* Must be timezone-aware
* Must be generated by backend

#### Must Not

* Use client-provided timestamps
* Use naive datetime objects
* Mix timezones

#### Risk

* Incorrect HOT classification due to time drift

---

### 1.7 State Management Constraints

#### Must

* State MUST always be derived from events
* `lead_signal` MAY be persisted as cache only

#### Must Not

* Treat stored state as source of truth
* Persist computed fields as authoritative data

#### Trade-off

* Slight computation overhead vs guaranteed correctness

---

### 1.8 API Constraints

#### Must

* All endpoints MUST be stateless
* All endpoints MUST validate input strictly
* All endpoints MUST return explicit success/failure

#### Must Not

* Perform blocking external calls
* Execute heavy computations

---

### 1.9 Performance Constraints

#### Must

* API latency ≤ 500ms (p95 < 800ms)
* State computation MUST remain lightweight

#### Must Not

* Execute full-table scans
* Perform unindexed queries
* Use blocking I/O

#### Enforcement

* Indexed queries
* Async processing

#### Escalation Trigger

* Latency > 800ms (p95)

---

### 1.10 Database Constraints

#### Must

* PostgreSQL for production
* Connection pooling enabled (pool_size=10, max_overflow=5)
* Indexed columns:

  * lead_id
  * section
  * occurred_at

#### Must Not

* Use unindexed queries in critical paths
* Allow schema drift

#### Data Constraints

* INSERT ONLY for `progress_events`
* No UPDATE / DELETE

---

### 1.11 Concurrency Constraints

#### Must

* System MUST remain consistent under concurrent requests
* Must rely on:

  * idempotency
  * append-only model
  * derived state

#### Must Not

* Use locking mechanisms that block requests
* Rely on last-write-wins logic

#### Model

* LAST-ACTIVITY-WINS (derived from timestamps)

---

### 1.12 External Integration Constraints (GHL)

#### Must

* Send HOT leads EXACTLY ONCE
* Use `ghl_synced` flag for enforcement
* Execute asynchronously

#### Must Not

* Block API requests
* Send duplicate data

#### Risk

* Duplicate lead creation in external system

---

### 1.13 Rate Limiting Constraints

#### Must

* Limit requests to 10 req/sec per IP

#### Must Not

* Allow unbounded request bursts

#### Enforcement

* Middleware or nginx

---

### 1.14 Logging Constraints

#### Must

* Log all:

  * API requests
  * Progress events
  * Errors

#### Must Not

* Omit critical failures
* Log sensitive data beyond email

---

### 1.15 Security Constraints

#### Must

* Validate all inputs
* Use HTTPS in production

#### Must Not

* Expose database directly
* Trust client input blindly

---

## 2. Preferences

* FastAPI (async) for backend
* PostgreSQL for persistence
* SQLAlchemy for ORM
* Nginx for reverse proxy + rate limiting
* UTC as global time standard

---

## 3. Trade-offs

| Decision             | Trade-off                   |
| -------------------- | --------------------------- |
| Deterministic rules  | No adaptive intelligence    |
| Append-only events   | Increased storage usage     |
| Derived state        | Slight compute overhead     |
| Async external calls | Increased system complexity |
| Strict constraints   | Reduced flexibility         |

---

## 4. Escalation Triggers

Immediate escalation required if:

* API latency > 800ms (p95)
* Duplicate events detected
* GHL duplicate sync occurs
* State mismatch detected
* Timezone inconsistency found
* DB index missing on critical query

---

## 5. Definition of Done

* [ ] All constraints enforced in codebase
* [ ] No event mutation possible
* [ ] Idempotency guaranteed at DB level
* [ ] Section ordering strictly registry-based
* [ ] TOTAL_SECTIONS defined in single location
* [ ] All timestamps verified as UTC
* [ ] API meets latency requirements
* [ ] Database indexes implemented
* [ ] Rate limiting active
* [ ] Async processing implemented for external calls
* [ ] Logging covers all critical paths
* [ ] System passes concurrency tests without inconsistency

---

## 6. Summary

These constraints define the **non-negotiable contract** of the system.

They ensure:

* Determinism
* Reliability
* Performance
* Consistency

Any violation of these constraints invalidates system correctness and MUST be treated as a critical defect.
