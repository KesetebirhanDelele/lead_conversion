# spec/00_overview.md

## 1. System Overview

The Lead Conversion System is a deterministic, event-driven backend integrated with a GPT-based learning interface. It tracks user progress through a structured course and classifies leads as `HOT` or `NOT_HOT` based on engagement.

The system is designed to:

* Capture user progress as immutable events
* Compute lead engagement deterministically
* Expose real-time state via API
* Trigger downstream systems when conversion criteria are met

This is **not an ML system**. All behavior is governed by explicit rules.

---

## 2. Core Objective

To qualify leads based on structured learning engagement and identify high-intent users for sales follow-up.

---

## 3. Key Definitions

### Lead

An individual identified by a unique email address who interacts with the GPT learning system.

```json
{
  "lead_id": "email@example.com"
}
```

---

### Progress Event

An immutable record indicating that a lead completed a specific course section.

```json
{
  "event_id": "lead_id:section",
  "lead_id": "email@example.com",
  "section": "P1_S1",
  "timestamp": "UTC ISO-8601"
}
```

---

### Conversion (MVP Definition)

A lead is considered converted (`HOT`) if:

* `completion_pct >= 25`
* `last_activity_at <= 48 hours from now`

---

### Derived State

Computed dynamically from progress events:

```json
{
  "completion_pct": 33.33,
  "current_section": "P2_S1",
  "last_activity_at": "2026-01-01T10:00:00Z",
  "lead_signal": "HOT"
}
```

---

## 4. System Scope

### Included

* GPT-based course interaction
* Progress tracking via API
* Deterministic state computation
* Lead classification (HOT / NOT_HOT)
* PostgreSQL-backed persistence
* Optional integration with GoHighLevel (GHL)
* Minimal dashboards (user + admin)

---

### Excluded

* Machine learning models
* CRM automation workflows
* Marketing campaign orchestration
* Multi-course support
* Complex personalization

---

## 5. High-Level Flow

1. User enters GPT interface
2. User provides email (lead_id)
3. User completes course sections sequentially
4. Each completion triggers a progress event via API
5. Backend stores event (idempotent, append-only)
6. Backend recomputes derived state
7. Lead is classified:

   * `HOT` if threshold met
   * `NOT_HOT` otherwise
8. If `HOT` and not previously synced:

   * Trigger external system (GHL)

---

## 6. Deterministic Core Principles

### 6.1 Single Source of Truth

* `TOTAL_SECTIONS` is defined only in:
  `execution/course/course_registry.py`

```python
from execution.course.course_registry import TOTAL_SECTIONS
```

---

### 6.2 Explicit Section Ordering

Section progression is strictly defined:

```python
COURSE_SECTIONS = [
    "P1_S1", "P1_S2", "P1_S3",
    "P2_S1", "P2_S2", "P2_S3",
    "P3_S1", "P3_S2", "P3_S3"
]
```

Rules:

* No sorting
* No string comparison
* Only index-based progression

---

### 6.3 Time Consistency

All timestamps:

* Must be UTC
* Must be timezone-aware
* Must be generated by backend

```python
from datetime import datetime, timezone
now = datetime.now(timezone.utc)
```

---

### 6.4 Event Model

* Events are append-only
* Events are immutable
* No updates or deletes allowed

---

### 6.5 Idempotency

Each event is uniquely identified:

```
event_id = lead_id + ":" + section
```

Duplicate submissions must be ignored.

---

### 6.6 Derived State Only

System state is never stored as source of truth.

All state must be computed from events:

```
state = f(progress_events)
```

Exception:

* `lead_signal` is persisted as a **cache/optimization**, not source of truth

---

## 7. System Components

### 7.1 GPT Interface

* Drives user interaction
* Calls backend APIs
* Does not store state

---

### 7.2 Backend (FastAPI)

Responsibilities:

* Validate inputs
* Store progress events
* Compute derived state
* Classify leads
* Expose APIs

---

### 7.3 Database

* MVP: SQLite
* Production: PostgreSQL

Stores:

* Leads
* Progress events

---

### 7.4 Lead Intelligence Engine

Implements deterministic rules:

```
completion_pct = completed_sections / TOTAL_SECTIONS * 100

is_recent = (now - last_activity_at) <= 48 hours

if completion_pct >= 25 and is_recent:
    lead_signal = "HOT"
else:
    lead_signal = "NOT_HOT"
```

---

### 7.5 External Integration (Optional)

* GoHighLevel (GHL)
* Triggered only once per HOT lead
* Controlled via `ghl_synced` flag

---

## 8. Non-Goals

The system explicitly avoids:

* Probabilistic scoring
* Heuristic ranking
* ML-based prediction
* Implicit behavior

---

## 9. Assumptions

ASSUMPTION: The course contains a fixed number of sections defined in a registry module
Alternative: Dynamic course definition (not supported in MVP)

ASSUMPTION: Leads are uniquely identified by email
Alternative: UUID-based identity system

ASSUMPTION: Single course per system
Alternative: Multi-course architecture (future)

---

## 10. Success Criteria

The system is considered successful if:

* ≥ 95% of valid progress events are recorded without duplication
* State computation latency < 100ms per request (p95)
* API availability ≥ 99.9%
* Lead classification is 100% deterministic (no variance)
* ≥ 90% of HOT leads are correctly identified based on rules

---

## 11. Risks & Mitigations

### Risk: Drift in section definitions

Mitigation: Single source of truth in `course_registry.py`

---

### Risk: Time inconsistency

Mitigation: Enforce UTC-only timestamps at backend level

---

### Risk: Duplicate events

Mitigation: Enforce unique `event_id`

---

### Risk: Premature optimization of state storage

Mitigation: Always derive state from events

---

## 12. Summary

This system is a **deterministic lead qualification engine** built on:

* Immutable event tracking
* Explicit progression rules
* Real-time state computation

Its strength lies in:

* Predictability
* Testability
* Zero ambiguity

All future extensions must preserve these properties.
